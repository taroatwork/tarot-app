## 1. 完成したアプリの概要

大アルカナ22枚を使った、ブラウザだけで完結するタロット占いアプリ。

### 主な機能
- **スプレッド3種**：ワンオラクル（1枚）／スリーカード（過去・現在・未来）／ケルト十字（10枚）
- **22枚のデッキ全体から選ぶ**：シャッフル→カット→裏向きカードをタップして必要枚数を選択
- **正位置／逆位置**：50%でランダム反転。絵柄も180°回転し、意味も反転
- **オリジナルのカード絵柄**：著作権に配慮し、各カードを固有色＋象徴シーン（太陽・月・柱・水・炎・運命の輪・山並み・星）でSVG描画
- **スプレッド本来の配置**：ケルト十字は「左の十字＋右の杖」の正式レイアウト
- **総合リーディング**：質問に紐づけた「今の中心／これからの流れ／結論／次の一歩」の4セクション解説
- **吉凶の色分け**：良い兆し（緑）／注意・課題（オレンジ）／中立（グレー）。逆位置だと色も反転
- **演出と効果音**：ゆっくりめくるフリップ、シャッフル／配り／選択／めくりの効果音（Web Audioで合成）、ミュート切替
- **履歴保存**：結果を端末内（localStorage）に保存・再表示・削除
- **配布用QRコード**：トップページに公開URLのQRを表示
- **ユニバーサルデザイン**：文字サイズとコントラストを高め、年配の方にも読みやすく

### 設計上の方針
- **完全オフライン・サーバー不要**：通信なし、外部APIなし、データは端末内のみ
- **単一HTMLファイル**：ビルド工程なし。`index.html` 1枚で動く静的サイト
- **著作権配慮**：既存タロットデッキの画像は一切使わず、すべてオリジナルのCSS/SVGで描画

## 2. 技術スタック（用語集）

このアプリで使った技術を、用語解説レベルでまとめます。**実装そのものはClaude Codeが行う**ため、初心者は「何のための道具か」を知っておけば十分です。

| 用語 | これは何か | このアプリでの役割 |
|------|-----------|------------------|
| **HTML** | Webページの骨組み（見出し・ボタン・文章などの構造）を書く言語 | 画面の要素を定義 |
| **CSS** | 見た目（色・大きさ・配置・アニメーション）を指定する言語 | フェルト背景、カード枠、フリップ演出、文字サイズなど |
| **JavaScript（バニラJS）** | ページに動き・ロジックを与えるプログラミング言語。「バニラ」はライブラリを使わない素のJSの意味 | シャッフル、カード選択、リーディング生成、画面遷移 |
| **SVG** | 拡大しても劣化しない図形（ベクター画像）の仕組み | カードの絵柄、QRコードの描画 |
| **localStorage** | ブラウザに少量のデータを保存する仕組み（端末内に残る） | 占い履歴・サウンド設定の保存 |
| **Web Audio API** | ブラウザで音をその場で合成・再生する仕組み | 効果音（音声ファイル不要） |
| **Tabler Icons** | 無料のアイコン集（Webフォントとして読み込む） | 各種アイコン（✦、時計、音量など） |
| **Google Fonts（Noto Sans/Serif JP）** | Webで使える日本語フォント | 明朝体の見出しなど |
| **qrcode / jsqr** | QRコードを「生成」するライブラリと「読み取り（デコード）」するライブラリ | QR生成と、その正しさの検証 |
| **Git / GitHub** | コードの変更履歴を管理し、クラウドに保存・公開する仕組み | バージョン管理・ソース公開 |
| **Vercel** | GitHubと連携して、静的サイトを自動で公開（デプロイ）するサービス | 本番公開・`git push` で自動再デプロイ |
| **CDN** | ライブラリやフォントを高速配信する外部サーバー | Tabler Icons・Google Fonts の読み込み |

> 💡 学びのポイント：**「単一HTMLファイル＋CDN」**という構成は、ビルドツールもサーバーも不要で、初心者が最初に成功体験を得るのに最適です。Vercelは「設定ほぼゼロ」で公開できます。

## 3. AI駆動開発の進め方

このプロジェクトで実践した、AIと進める開発の基本サイクルです。

```
①相談・仕様決め → ②AIが生成 → ③実機で確認 → ④具体的にフィードバック → ②へ戻る（反復）
```

### うまくいった「指示の出し方」のコツ
- **最初に仕様を一緒に詰める**：いきなり「作って」ではなく、「仕様を考えて」と頼み、選択肢から選んで合意してから着手した。
- **判断はAIに委ねず、要所は自分で決める**：枚数（大アルカナ22枚）、AI解釈の有無（内蔵テキスト）など、分岐点はユーザーが選択した。
- **実機で見て具体的に指摘する**：「QRが大きすぎる」「文字が小さくて読めない」「左の列が窮屈」など、**現象＋理由**で伝えると修正が的確になる。
- **AIに意見を求める**：「あなたはどう思う？」と問うと、AIが根拠つきで提案を返し、判断材料が増える。
- **ベストプラクティスを調べさせる**：「タロットアプリのベストプラクティスを調べた上で直して」と頼み、Web検索の結果を設計に反映させた。
- **AIの正直さを引き出す**：QRコードの件のように「本当に自信があるか？」と確認し、AIに能力の限界（手描きは不可）と確実な方法（ライブラリ＋検証）を線引きさせた。

> 💡 学びのポイント：AI駆動開発では、**人間の役割は「方向づけ」と「品質判定」**に移ります。コードを書く力よりも、**何を作るか・何が良し悪しか**を言語化する力が効いてきます。

## 4. 開発の時系列ログ

各ステップを「🗣️ 指示原文 → 🛠 対応 → 💡 学び」で記録します。

### ステップ0：企画と仕様決め

🗣️ **指示原文**
> 君さ、タロットカードアプリみたいなものできる？あなたのキャンバスの上で動くウェブアプリとして開発したいんですよ。ちょっと仕様を考えてもらえませんか。

🛠 **対応**
- いきなり実装せず、仕様のたたき台を提示。スプレッド種別、カードデータ、デザイン方針、技術構成を整理。
- 「カードの枚数」「AI解釈の有無」「進め方」を質問し、ユーザーが選択 → **大アルカナ22枚／内蔵テキスト／仕様を詰めてから**に決定。
- 仕様書 v1.0（画面フロー・データ構造・著作権配慮方針）を提示して合意。

💡 **学び**：作り始める前に**仕様を文書化して合意**すると、後の手戻りが激減する。

---

### ステップ1：初版（キャンバス上で動くプロトタイプ）

- 合意した仕様で、Claudeのキャンバス（チャット内で動くインタラクティブHTML）に動くプロトタイプを生成。
- この時点では絵柄は簡素（アイコン1個）、リーディングは各カードの意味の羅列。

💡 **学び**：まず**動くもの**を早く出して、画面を見ながら議論する方が早い。

### ステップ2：絵柄とリーディングの実装

🗣️ **指示原文**
> 君が作ってくれたこのアプリだけど、カードを選んで内容が表示されるだけで、絵柄も出ないし、僕の質問に対する回答も出てこない。そこのとこ、よくタロットカードアプリのベストプラクティスを調べた上で、ちゃんと動くように修正してください。

🛠 **対応**
- Web検索で「タロットの読み方」「3枚引きの統合解釈」などのベストプラクティスを調査。
  - カード個別の意味の羅列ではなく、**カード同士をつないだ物語（シンセシス）**で質問に答えるのが定石。
  - 大アルカナは「人生の大きな節目」を示す最重要エネルギーとして扱う。
- 各カードに**固有色＋象徴シーン**のオリジナルSVG絵柄を実装（逆位置は180°反転）。
- 質問に紐づく**総合リーディング**を内蔵テキストから組み立てるロジックを追加。

💡 **学び**：**「ベストプラクティスを調べてから」**と一言添えるだけで、出力の質が大きく上がる。

---

### ステップ3：全22枚から選ぶ／ケルト十字の正式配置／読みの具体化

🗣️ **指示原文**
> 22枚全部の中から10枚を選ぶような画面デザインになっていないですよね。…ケルト十字の場合だったら…きちんと配置した上でユーザーに見せるようなUI/UXが必要じゃないかな…出てくる読み解きなんですけど、ちょっと抽象度が高すぎる…もうちょっとわかりやすい表現に…

🛠 **対応**
- 引く画面を**22枚すべて**を扇状に並べる方式に変更。
- ケルト十字を**「左の十字＋右の杖」の正式配置**に（位置番号バッジ付き）。スリーカードは横一列、ワンオラクルは中央。
- 読みを**「結論＋次の一歩」**型に再構成し、カード個別の意味文も日常的な表現に全面改訂。吉凶を点数化して質問への傾きを判定。

💡 **学び**：**「配置」そのものが意味を持つ**（ケルト十字）。見た目＝仕様であり、レイアウトは機能の一部。

---

### ステップ4：4セクション化・吉凶の色分け・正逆の視覚化・配置解説

🗣️ **指示原文（要旨を含む）**
> …4つのセクション…をもっと詳しく…ただし長すぎると読み手としてつらい…優秀な占い師がどれぐらいの言葉の量で説明するのか…／ポジティブ・ネガティブ・中立…が視覚的に色分け…／正位置と逆位置で意味が反転…視覚的にも分かりやすく…／左側と右側でどのような意味があるのか…解説も…

🛠 **対応**
- リーディングを「今の中心／これからの流れ／結論／次の一歩」の**独立セクション**に分割（各2〜3文の分量に調整）。
- 吉凶を**3段階で色分け**（緑／オレンジ／グレー）。**逆位置だと色も反転**。
- 逆位置は絵柄反転＋下向き矢印、「正位置なら○○ → 逆位置で反転」の対比表示。
- ケルト十字に**「配置の意味」解説**（左の十字／右の杖）と各位置の説明を追加。

💡 **学び**：「分かりやすく」は曖昧なので、**「優秀な占い師ならどれくらいの分量か」**のように**基準を与える**と精度が上がる。

### ステップ5：GitHub公開とVercelデプロイ

🗣️ **指示原文**
> このアプリケーションを私のGitHubリポジトリにプッシュした上で、Vercelを使ってデプロイしたいと思います。どのような手順で進めたらよいですか。

🛠 **対応**
- **重要な変換**：キャンバスのウィジェットはホスト（Claude）依存（CSS変数・アイコン・フォント）。そのままでは外部で動かない。→ それらを**自前で同梱した単一HTML**に変換。
- ダークな神秘的テーマのCSS変数を自前定義、Tabler Icons・Google FontsをCDN読み込み、ページ背景を追加。
- `git init` → コミット → `gh repo create`（GitHub CLI）でリポジトリ作成＆プッシュ。
- Vercelは Framework Preset = **Other**、Build/Output 空のまま（静的サイト）でデプロイ。
- 2FA設定ダイアログやImport画面の各項目も、画面を見ながら逐一案内。

💡 **学び**：**「動く環境」と「配布する環境」は別物**。依存関係を持ち出す（自己完結化する）作業が必要になることがある。

### ステップ6：背景デザイン（モスグリーンのフェルト）

🗣️ **指示原文**
> 背景の色とカードの色が同じなので、見にくいです。…カードテーブル（家具）のような、モスグリーンのフェルトのような背景も良いかなと思います。

🛠 **対応**：中央が明るく縁が暗い陰影＋微細な織り目テクスチャの**フェルト調背景**に変更。深紫のカードが映えるように。

💡 **学び**：背景と前景の**コントラスト**は視認性の基本。実際の素材（フェルト）をイメージに使うと指示が明確になる。

### ステップ7：ケルト十字の間隔とモバイル対応

🗣️ **指示原文（2件）**
> 左の十字の横の列ですが、カードの間隔が狭くて、見にくいです。調整してください。

> iPhoneで見ると横スクロールが必要になっています。モバイルの場合は横スクロールなしで表示できるように調整してください。

🛠 **対応**
- 横の列の間隔を拡大（中央で90度回転して張り出す②のカードと左右が重ならないように）。
- ケルト十字全体を**画面幅に自動フィット**（はみ出す分だけ `transform: scale()` で縮小、画面回転にも追従）。

💡 **学び**：**実機確認**は必須。PCでは問題なくてもモバイルで破綻することが多い（横スクロール、文字サイズ等）。

---

### ステップ8：QRコード（手描き不可・ライブラリ＋検証）

🗣️ **指示原文**
> 君は正しいQRコードを生成することに自信がありますか？以前…自信満々だったけど、生成されたQRコードはつかいものにならないことがありました。今のあなたはどうなんだろう。

🛠 **対応（正直な線引き）**
- 「**手描き（モジュールを自分で配置）は不可**」と明言。QRは誤り訂正符号・マスク選択など厳密な規格で、暗記・手作業では破綻する。
- 確実な方法：`qrcode` ライブラリで生成 → `jsqr` で**生成物をデコードしてURLに戻るか検証**。
- 実際に `MATCH: true`（完全一致）を確認し、検証済みSVGをトップに埋め込み。埋め込み後も**バイト単位で一致**を再確認。

💡 **学び（最重要）**：AIには**得意・不得意**がある。**「自分で描く」のではなく「正しい道具に作らせ、結果を検証する」**のが信頼できるやり方。AIに正直な自己評価を求めるのは有効。

### ステップ9：トップページUIの磨き込み

複数の小さな指示を反復で反映。いずれも**実機を見ての具体的な指摘**。

- QRが大きすぎる → 配置・サイズを段階的に調整（最終的に72px）。
- 主要ボタンの主張が弱い →「**タロットカードの背面デザイン**にしては？」という提案を受け、**縦長のカード型ボタン**（カード＋下にラベル）に。
- QRキャプションが左寄せ・改行で崩れる → **順序を入れ替え**（説明文を上、QRを最下部に）、QRとキャプションを**センタリング**。
- サウンドボタンを**タイトル上のダイヤと同じ高さ**に移動。
- ボタン下の余白が広すぎる → 余白を詰めて引き締め。

💡 **学び**：UIは**反復**で仕上がる。一度で完璧を狙わず、見て直すを繰り返す。AIは何度でも即座に直せるのが強み。

---

### ステップ10：演出と効果音

🗣️ **指示原文**
> このカードをめくるアニメーションのスピードが速すぎる…実際の占い師はもっともったいぶってゆっくりとめくります。…サウンドエフェクトも重要…シャッフルする。…広げる。…めくる。それぞれ…適切なサウンドエフェクトが…出力されると気分が高まります。

🛠 **対応**
- フリップを**1.35秒**に、1枚ずつの間隔も拡大（10枚のときは長くなりすぎないよう調整）。
- 効果音を**Web Audioでその場合成**（音声ファイル不要）：シャッフル／配り／選択／めくり。
- **ミュート切替**を追加（設定は保存）。ブラウザの自動再生制限に対応し、最初のタップ以降に有効化。

💡 **学び**：体験（UX）は**間（ま）と音**で大きく変わる。効果音は外部ファイルに頼らずコードで合成でき、オフライン配信にも強い。

---

### ステップ11：リーディングの表示タイミング（めくり後に表示）

🗣️ **指示原文**
> めくりのアニメーションに時間をかけたことによって、読み取りの結果が最初から表示されてしまっている…取りはすべてのカードが表向きにされてから行われるもので…すべてのカードをめくり終わった後に、もったいぶって徐々に表示される…良いUXにつながると思います。

🛠 **対応**：めくり中はリーディングを非表示にし、全カードめくり終了後に「総合リーディング → 各カードの意味 → ボタン」を**段階的にフェードイン**。

💡 **学び**：**情報の出る順序**もUX。タロットの作法（全部めくってから読む）を尊重することで没入感が増す。

---

### ステップ12：モバイルでのヒント可視性

🗣️ **指示原文**
> ケルト十字の場合、…解説のカードが最初から表示されているので、…カードを1枚ずつめくっていますとか…のテキストメッセージはスクロールしないと見ることができません。これは全く意味がない…

🛠 **対応**：ヒント文を**カードの上**へ移動し、配置解説は結果と一緒に（めくり後に）表示。

💡 **学び**：要素の**画面内の位置**は、特にモバイルで決定的。「表示している＝見えている」とは限らない。

---

### ステップ13：ユニバーサルデザイン（文字サイズ・コントラスト）

🗣️ **指示原文**
> ヒント文…フォントサイズがかなり小さく、背景とのコントラストが低く、63歳の僕の目にははっきり読むことができません。…アプリケーション全体のユニバーサルデザインを改めて考えて、UIデザインの修正を…

🛠 **対応**
- 補助テキストの色を**全体的に明るく**（コントラスト向上）。
- **11〜12pxの極小文字を13〜15pxに底上げ**（最小13px方針）。ヒント文は金色・19px・太字に。

💡 **学び**：**実ユーザーの身体的条件**（年齢・視力）からの指摘は最良のフィードバック。アクセシビリティは後付けでなく品質の中核。

### ステップ14：バッジ／ラベルの表示タイミング（最後の修正）

🗣️ **指示原文**
> カードの左肩に表示される丸いアイコンの表示タイミング…今はカードがめくられる前から表示されていますが、本来…読み取ることができるのは、カードがめくられた後…カードを1枚めくるごとに…順に表示していくのが正しい…

🛠 **対応**：番号・吉凶バッジ（ケルト十字）と正逆・吉凶ラベル（スリーカード/ワンオラクル）を初期非表示にし、**各カードがめくられた瞬間（フリップ中盤、表面が見え始める頃）に1枚ずつフェードイン**するよう統一。

💡 **学び**：**情報の整合性**——「まだ分からないはずの情報」を先に見せない。一貫した世界観がUXの説得力を生む。

## 5. つまずき＆気づき集（ハイライト）

セミナーで特に強調したい教訓。

1. **キャンバスのウィジェット ≠ 配布できるアプリ**
   ホスト環境に依存していたものを、自己完結した単一HTMLに**変換**する必要があった。「動いている」と「配れる」は別。

2. **QRコードは手描き不可、ライブラリ＋検証が正解**
   AIに過信は禁物。**正しい道具に作らせ、出力を機械的に検証**する。AIに正直な自己評価を求めると、安全な進め方が見える。

3. **実機確認が全てを決める**
   横スクロール、文字サイズ、余白、表示位置——PCでは気づけない問題がモバイルで噴出した。

4. **ユニバーサルデザインは品質の中核**
   63歳のユーザー自身の「読めない」という一言が、全体の文字サイズ・コントラスト見直しにつながった。

5. **UXは「順序・間・音・整合性」で決まる**
   めくり速度、結果を出す順番、効果音、バッジの出現タイミング——機能ではなく**体験**を作り込むほど満足度が上がる。

6. **AIへの指示は「現象＋理由＋希望」で**
   「QRが大きすぎる（現象）。低密度だから半分でも読めるはず（理由）。半分にして（希望）」のように伝えると、AIの対応が的確になる。

## 6. GitHub / Vercel デプロイ手順（再現用）

別アプリでも使える、最小手順。

### 前提ツール
- `git`、`gh`（GitHub CLI）、必要に応じて `node`（QR生成などの検証用）
- GitHubアカウント、Vercelアカウント

### 手順
```bash
# 1. プロジェクトフォルダで初期化
cd <プロジェクト>
git init
git add -A
git commit -m "Initial commit"

# 2. GitHubにリポジトリを作成してプッシュ（gh CLI）
gh auth status                 # 未ログインなら gh auth login
gh repo create <リポジトリ名> --public --source=. --remote=origin --push

# 3. Vercelで公開（ダッシュボード）
#   - https://vercel.com/new を開く
#   - 作成したリポジトリを Import
#   - Framework Preset = Other（静的サイト）
#   - Build Command / Output Directory は空のまま
#   - Deploy
```

- 以降は `git push` するたびに**自動で再デプロイ**される。
- QRコードを安全に作るなら：`qrcode` で生成 → `jsqr` でデコードして元URLに戻るか**検証**してから埋め込む。

> 💡 学びのポイント：静的サイトはVercelで**ほぼ設定ゼロ**で公開できる。最初の一本はこの構成が最も挫折しにくい。

## 7. 演習課題

セミナー受講者向けの実習アイデア。同じ「対話で作る → GitHub → Vercel」の流れを再現する。

   - 小アルカナ56枚を追加してフル78枚デッキにする
